# Implementation Spec: Dark/Light Theme Toggle

## Summary

Add a dark/light theme toggle to the RPM portal. The app already has `next-themes` v0.4.6 installed, dark mode CSS variables fully defined in `globals.css`, and the Tailwind `@custom-variant dark` configured. The only missing pieces are: (1) wrapping the app in `ThemeProvider`, (2) adding a toggle UI, and (3) fixing inconsistent dark-mode sidebar CSS variables.

---

## Data Model

**No database changes required.** Theme preference is persisted client-side via `next-themes` (localStorage key: `theme`). No user table column needed.

---

## Available Data

- `next-themes` v0.4.6 is already in `package.json` (installed)
- `globals.css` already defines both `:root` (light) and `.dark` CSS variable sets
- `@custom-variant dark (&:is(.dark *))` is already configured in `globals.css` line 5
- All shadcn/ui components already use CSS variables (`bg-background`, `text-foreground`, etc.) so they will respond to theme changes automatically

## Proposed Implementation

### 1. Fix Dark Mode Sidebar CSS Variables

**File: `src/app/globals.css`** (lines 106-113)

The `.dark` block has inconsistent sidebar variables using `hsl()` while everything else uses `oklch()`. These should be converted to `oklch()` to match the light mode sidebar pattern and maintain the same dark navy aesthetic.

Replace:
```css
--sidebar: hsl(240 5.9% 10%);
--sidebar-foreground: hsl(240 4.8% 95.9%);
--sidebar-primary: hsl(224.3 76.3% 48%);
--sidebar-primary-foreground: hsl(0 0% 100%);
--sidebar-accent: hsl(240 3.7% 15.9%);
--sidebar-accent-foreground: hsl(240 4.8% 95.9%);
--sidebar-border: hsl(240 3.7% 15.9%);
--sidebar-ring: hsl(217.2 91.2% 59.8%);
```

With oklch equivalents that maintain the same dark sidebar look:
```css
--sidebar: oklch(0.15 0.03 240);
--sidebar-foreground: oklch(0.95 0.01 240);
--sidebar-primary: oklch(0.35 0.08 160);
--sidebar-primary-foreground: oklch(1 0 0);
--sidebar-accent: oklch(0.25 0.03 240);
--sidebar-accent-foreground: oklch(0.95 0.01 240);
--sidebar-border: oklch(0.25 0.03 240);
--sidebar-ring: oklch(0.45 0.1 160);
```

**Reasoning**: Keep the green branding consistent in dark mode sidebar (matching primary/accent hues from light mode) rather than switching to blue hsl values that don't match the brand.

### 2. Add ThemeProvider to Root Layout

**File: `src/app/layout.tsx`**

This is a **server component** (no `"use client"` directive), so the ThemeProvider must be wrapped in a client component or the `ThemeProvider` itself must be imported properly.

**Approach**: Create a thin `ThemeProvider` wrapper component.

**New file: `src/components/theme-provider.tsx`**
```tsx
"use client"

import { ThemeProvider as NextThemesProvider } from "next-themes"

export function ThemeProvider({
  children,
  ...props
}: React.ComponentProps<typeof NextThemesProvider>) {
  return <NextThemesProvider {...props}>{children}</NextThemesProvider>
}
```

This is the standard shadcn/ui pattern for wrapping next-themes. Reference: this is a well-established pattern used by shadcn/ui docs.

**Then modify `src/app/layout.tsx`**:

Add import:
```tsx
import { ThemeProvider } from "@/components/theme-provider"
```

Wrap the body content:
```tsx
<body className={`${geistSans.variable} ${geistMono.variable} antialiased`}>
  <ThemeProvider
    attribute="class"
    defaultTheme="light"
    enableSystem={false}
    disableTransitionOnChange
  >
    {children}
    <Toaster />
  </ThemeProvider>
</body>
```

**Key props**:
- `attribute="class"` — applies `.dark` class to `<html>`, which matches the existing `@custom-variant dark (&:is(.dark *))` and `.dark { }` CSS block
- `defaultTheme="light"` — task says "defaults to the current dark theme" but the app currently runs in light mode (no `.dark` class applied anywhere), so `"light"` is the correct default to match current behavior
- `enableSystem={false}` — keep it simple, explicit user choice only
- `disableTransitionOnChange` — prevents flash of unstyled content during theme switch

**Important**: The `<html>` tag needs `suppressHydrationWarning` added because next-themes modifies it client-side:
```tsx
<html lang="en" suppressHydrationWarning>
```

### 3. Create Theme Toggle Component

**New file: `src/components/theme-toggle.tsx`**

A simple toggle button that cycles between light and dark themes.

```tsx
"use client"

import { useTheme } from "next-themes"
import { Moon, Sun } from "lucide-react"
import { Button } from "@/components/ui/button"

export function ThemeToggle() {
  const { theme, setTheme } = useTheme()

  return (
    <Button
      variant="ghost"
      size="icon"
      onClick={() => setTheme(theme === "dark" ? "light" : "dark")}
      className="size-9"
    >
      <Sun className="size-4 rotate-0 scale-100 transition-all dark:rotate-90 dark:scale-0" />
      <Moon className="absolute size-4 rotate-90 scale-0 transition-all dark:rotate-0 dark:scale-100" />
      <span className="sr-only">Toggle theme</span>
    </Button>
  )
}
```

This is the standard shadcn/ui theme toggle pattern using Lucide icons (already installed). The Sun/Moon icons animate on transition using Tailwind `dark:` variant classes.

## Risks & Dependencies

1. **Hydration mismatch**: `next-themes` modifies the `<html>` element client-side. The `suppressHydrationWarning` prop on `<html>` and `disableTransitionOnChange` on `ThemeProvider` mitigate this. This is the standard approach.

2. **Leaflet map styling**: The map component (`store-map.tsx`) uses OpenStreetMap tiles which have their own styling. The map itself won't change with dark mode, but surrounding UI will. This is acceptable — map tile theming is out of scope.

3. **Chart colors**: Recharts components in the analysis page use `chart-1` through `chart-5` CSS variables. These are **not currently defined** in either `:root` or `.dark` blocks in `globals.css`. If charts look wrong in dark mode, chart color variables may need to be added. **This is a known gap but out of scope for this task** — charts will use whatever default colors Recharts provides.

4. **Flash of incorrect theme (FOIT)**: `next-themes` handles this via a script injected before React hydration. The `attribute="class"` approach combined with `disableTransitionOnChange` prevents visible flicker.

5. **Sidebar in dark mode**: The sidebar already uses dark colors in light mode (`oklch(0.2 0.05 240)`). In dark mode, the sidebar should remain dark but the colors need to be consistent. The fix in step 1 ensures this.

6. **`useTheme` returns undefined on first render**: `next-themes` `useTheme()` returns `undefined` for `theme` during SSR/first render. The theme toggle should handle this gracefully — the Sun/Moon animation approach using CSS `dark:` classes handles this because it doesn't depend on the JS `theme` value for icon display. For the settings page buttons, the `variant` comparison with `undefined` will just show both as `outline` initially, which is fine.
