## Issues Identified

### Issue 1: No `get_my_client_id()` function
Client isolation policies query the `users` table directly to get `client_id`. Since `users` has RLS, this creates recursion risk. Need a SECURITY DEFINER function like `get_my_role()`.

**Affected policies:**
- `client_hq_view` on `clients` (line 70-76 of `20260316000000`)
- `client_isolation_stores` on `stores` (line 90-96 of `20260316000000`)
- `client_isolation_assets` on `assets` (line 31-37 of `20260316100000`)
- `client_isolation_jobs` on `jobs` (line 51-57 of `20260316100000`)
- `vendors_select` on `vendors` (line 97-99 of `20240305000000`)
- `vendors_insert` on `vendors` (line 101-103 of `20240305000000`)
- `projects_select` on `projects` (line 106-108 of `20240305000000`)
- `projects_insert` on `projects` (line 110-112 of `20240305000000`)
- `regions_select` on `regions` (line 115-117 of `20240305000000`)
- `regions_insert` on `regions` (line 119-121 of `20240305000000`)
- `site_photos_select` on `site_photos` (line 124-130 of `20240305000000`)
- `asset_photos_select` on `asset_photos` (line 133-140 of `20240305000000`)
- `maintenance_schedules_select` on `maintenance_schedules` (line 143-150 of `20240305000000`)

### Issue 2: Overly broad GRANTs
All tables get full CRUD GRANTs. Since all authenticated users share the `authenticated` PostgreSQL role, we cannot restrict GRANTs per user â€” RLS is the real enforcement layer. However, for tables where NO app user ever needs certain operations, we can remove those GRANTs as defense-in-depth.

**Analysis of needed operations per table:**

| Table | SELECT | INSERT | UPDATE | DELETE | Rationale |
|-------|--------|--------|--------|--------|-----------|
| `users` | Yes | Yes | Yes | Yes | Admins create/edit users, users edit self |
| `clients` | Yes | Yes | Yes | Yes | Super admins manage clients |
| `stores` | Yes | Yes | Yes | Yes | Admins manage stores |
| `assets` | Yes | Yes | Yes | Yes | Admins/techs manage assets |
| `jobs` | Yes | Yes | Yes | Yes | Any user reports faults, admins manage |
| `asset_types` | Yes | Yes | Yes | Yes | Admins manage catalog via UI |
| `vendors` | Yes | Yes | Yes | Yes | Admins manage vendors via UI |
| `regions` | Yes | Yes | Yes | Yes | Admins manage regions via UI |
| `maintenance_schedules` | Yes | Yes | Yes | Yes | Admins manage schedules |
| `site_photos` | Yes | Yes | No | Yes | Photos are inserted/deleted, never updated |
| `asset_photos` | Yes | Yes | No | Yes | Photos are inserted/deleted, never updated |

### Issue 3: `asset_types` has no RLS
The `asset_types` table was created in the initial schema but RLS was never enabled. Any authenticated user can INSERT/UPDATE/DELETE asset types.

### Issue 4: Missing write RLS policies
Many tables only have SELECT policies. Without INSERT/UPDATE/DELETE policies, RLS blocks ALL write operations for non-admin users (admins bypass via `FOR ALL` policies on some tables, but not all).

### Issue 5: Analysis page error handling
File: `src/app/analysis/page.tsx`
- Lines 100, 119, 130: Errors are `console.error`'d but not surfaced to the user
- Line 279-286: Generic "Unable to load analytics data" message with no detail
- No error state variable exists

## Proposed Implementation

### File Changes

#### 1. New Migration: `supabase/migrations/20260317000000_fix_rls_security.sql`

This is the core deliverable. One migration file that:

**Step 1: Create `get_my_client_id()` SECURITY DEFINER function**
```sql
CREATE OR REPLACE FUNCTION public.get_my_client_id()
RETURNS uuid
LANGUAGE sql
STABLE
SECURITY DEFINER
SET search_path = public
AS $$
  SELECT client_id FROM users WHERE id = auth.uid();
$$;

GRANT EXECUTE ON FUNCTION public.get_my_client_id() TO authenticated;
GRANT EXECUTE ON FUNCTION public.get_my_client_id() TO anon;
```

**Step 2: Fix `client_hq_view` on `clients` table**
```sql
DROP POLICY IF EXISTS "client_hq_view" ON clients;

CREATE POLICY "client_hq_view" ON clients FOR SELECT USING (
  id = public.get_my_client_id()
);
```
This replaces the subquery into `users` with a direct comparison using the SECURITY DEFINER function. Note: this policy now applies to ALL non-admin roles (technician, client_store, client_hq) â€” not just client_hq. This is correct behavior since all client-scoped users should see their own client record.

**Step 3: Fix `client_isolation_stores` on `stores` table**
```sql
DROP POLICY IF EXISTS "client_isolation_stores" ON stores;

CREATE POLICY "client_isolation_stores" ON stores FOR SELECT USING (
  client_id = public.get_my_client_id()
);
```

**Step 4: Fix `client_isolation_assets` on `assets` table**
```sql
DROP POLICY IF EXISTS "client_isolation_assets" ON assets;

CREATE POLICY "client_isolation_assets" ON assets FOR SELECT USING (
  EXISTS (
    SELECT 1 FROM stores
    WHERE stores.id = assets.store_id
    AND stores.client_id = public.get_my_client_id()
  )
);
```
Note: We still need to join through `stores` to get `client_id`, but we no longer touch the `users` table. The `stores` table's RLS won't cause recursion here because Supabase evaluates the policy's subquery with the caller's permissions â€” but `stores` RLS policies themselves use `get_my_role()` and `get_my_client_id()`, which are SECURITY DEFINER and skip RLS. So this is safe.

**Step 5: Fix `client_isolation_jobs` on `jobs` table**
```sql
DROP POLICY IF EXISTS "client_isolation_jobs" ON jobs;

CREATE POLICY "client_isolation_jobs" ON jobs FOR SELECT USING (
  EXISTS (
    SELECT 1 FROM stores
    WHERE stores.id = jobs.store_id
    AND stores.client_id = public.get_my_client_id()
  )
);
```

**Step 8: Add missing job INSERT policy for non-admin users**
Currently only admins can insert jobs (via `admin_view_jobs FOR ALL`). Non-admin users should be able to report faults.
```sql
CREATE POLICY "authenticated_insert_jobs" ON jobs FOR INSERT WITH CHECK (
  auth.role() = 'authenticated'
  AND EXISTS (
    SELECT 1 FROM stores
    WHERE stores.id = jobs.store_id
    AND stores.client_id = public.get_my_client_id()
  )
);
```

**Step 9: Scope GRANTs**

Revoke overly broad GRANTs first, then re-grant appropriately:
```sql
-- Revoke all first
REVOKE INSERT, UPDATE, DELETE ON site_photos FROM authenticated;
REVOKE INSERT, UPDATE, DELETE ON asset_photos FROM authenticated;

-- Re-grant without UPDATE (photos are never updated)
GRANT SELECT, INSERT, DELETE ON site_photos TO authenticated;
GRANT SELECT, INSERT, DELETE ON asset_photos TO authenticated;

-- Grant for newly RLS-enabled table
GRANT SELECT, INSERT, UPDATE, DELETE ON asset_types TO authenticated;
```

**Important note on GRANTs**: Since all authenticated users share the `authenticated` PostgreSQL role, we cannot use GRANTs to differentiate between user roles â€” that's what RLS does. Removing INSERT/UPDATE/DELETE GRANTs from a table would block ALL users including admins. The real security fix is ensuring every table has proper RLS policies with role-based restrictions (which steps 2-8 accomplish). The only safe GRANT reductions are removing operations no user ever performs (e.g., UPDATE on photo tables).

## Risks & Dependencies

### Risk 1: Policy conflicts with existing policies
The migration drops and recreates policies. If the migration runs on a database where the original (pre-fix) policies still exist (from initial schema), those old policies won't be dropped. The `DROP POLICY IF EXISTS` statements in the migration only target the policy names from the fix migrations.

**Mitigation**: Include `DROP POLICY IF EXISTS` for ALL known policy names (both from initial schema and fix migrations).

### Risk 2: RLS recursion in nested joins
Policies on `assets`, `jobs`, `site_photos`, `asset_photos`, `maintenance_schedules` join through `stores` to check `client_id`. The `stores` table has its own RLS. However, since `stores` RLS policies use `get_my_role()` and `get_my_client_id()` (SECURITY DEFINER functions that bypass RLS), this should not cause recursion.

**Verification needed**: Test that a non-admin user can query assets, jobs, and schedules without recursion errors.

### Risk 3: `get_my_client_id()` returns NULL for super_admin
Super admins have `client_id = NULL`. The `client_isolation_*` policies compare `client_id = get_my_client_id()`, which would be `client_id = NULL` â€” this evaluates to false in SQL. But super admins are handled by the separate `admin_*` FOR ALL policies which match first, so this is safe.

### Risk 4: Technicians and client_store users
The `client_isolation_*` policies use `get_my_client_id()` which works for any user with a `client_id` set. Technicians have `client_id` set, so they'll see their client's data. Store-level access filtering (`store_ids[]`) is handled at the application level, not RLS.

### Risk 5: Photo tables - existing INSERT operations
The `site-photo-gallery.tsx` and `asset-photo-gallery.tsx` components insert into `site_photos` and `asset_photos`. The new INSERT policies will require the user's `client_id` to match the store's `client_id`. Verify these components pass the correct `store_id` on insert.

## Out of Scope

- Implementing store-level access control via RLS (using `store_ids[]` array) â€” this is application-level filtering
- Changes to storage bucket policies (separate from table RLS)
- Adding RLS for the `projects` table INSERT/UPDATE/DELETE for non-admin users (projects are admin-managed)
- Modifying any pages other than the analysis page
- Creating database roles beyond `authenticated` (Supabase limitation)
