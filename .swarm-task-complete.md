# Review: ISSUE-RPM-6

## Verdict: NEEDS_CHANGES

### Summary

The core security fix (migration `20260317000000`) is well-structured and correctly addresses the RLS recursion issue by introducing `get_my_client_id()`. However, there are two issues: (1) the earlier migration `20260316000000` still grants overly broad INSERT/UPDATE/DELETE on nearly every table to all authenticated users, which the new migration only partially fixes (only photo tables are scoped), and (2) the analysis page was rewritten far beyond the scope of the task — the acceptance criteria only asked for "user-visible error state," but the entire page was replaced with a completely different analytics implementation.

### Issues

- **[severity: critical] Overly broad GRANTs remain unfixed.** Migration `20260316000000_fix_rls_policies.sql` (lines 101-111) grants `SELECT, INSERT, UPDATE, DELETE` on `users`, `clients`, `stores`, `assets`, `jobs`, `vendors`, `regions`, `maintenance_schedules` to all authenticated users. The new migration `20260317000000` only revokes excess grants on `site_photos` and `asset_photos` (lines 228-235). The acceptance criteria explicitly says "Scope GRANTs per table — most should only grant SELECT." Tables like `clients`, `regions`, `stores` should not grant INSERT/UPDATE/DELETE to all authenticated users — the RLS write policies would still block unauthorized writes, but defense-in-depth requires scoping the GRANTs too. At minimum, `clients` and `regions` should only grant SELECT to authenticated (admins get write via FOR ALL policies).

- **[severity: major] Analysis page rewritten beyond task scope.** The diff shows ~270 lines removed and ~615 lines added. The page was completely redesigned — new data model (asset health focus vs. job focus), new chart types (AreaChart, PieChart replacing LineChart), new interfaces, removed features (project budget stats, brand distribution, link to jobs). The task only required: "Add user-visible error state on analysis page." The error handling itself (lines 294-309) is good, but the surrounding rewrite is unauthorized scope creep.

- **[severity: minor] `get_my_client_id()` granted to `anon` role.** Line 21 of the security migration grants EXECUTE to `anon`. Since anonymous users have no `auth.uid()`, the function will always return NULL for them. This isn't a security risk (NULL won't match any client_id), but granting to `anon` is unnecessary and slightly widens the attack surface. Only `authenticated` needs this grant.

- **[severity: minor] Migration `20260316100000` still joins through users table.** The `client_isolation_assets` and `client_isolation_jobs` policies in `20260316100000_fix_assets_jobs_rls.sql` still use `JOIN users ON users.client_id = stores.client_id WHERE users.id = auth.uid()`. The new migration drops and recreates these policies using `get_my_client_id()`, which correctly fixes the issue — but if the new migration fails partway through, the old vulnerable policies from `20260316100000` could remain. Consider wrapping the new migration in a transaction (or verifying Supabase migrations are transactional by default).

### Suggestions

- The `fetchAnalytics` function reference on line 303 (`onClick={() => fetchAnalytics()}`) works but `fetchAnalytics` is defined with `const` inside the component — good, just verify it's hoisted correctly (it is, since it's a const assigned before the JSX return).
- Consider adding `REVOKE INSERT, UPDATE, DELETE ON clients FROM authenticated` and similar for `regions`, `users` (except self-update via RLS), to match the "scope GRANTs" acceptance criterion.
- The analysis page rewrite should be reverted to the previous version with only the error handling additions, or split into a separate task/PR.
