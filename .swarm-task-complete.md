# Swarm Task Complete: ISSUE-RPM-6

## What was done

Implemented RLS security fixes as specified in the implementation spec:

1. **Created `get_my_client_id()` SECURITY DEFINER function** — mirrors existing `get_my_role()` pattern, returns the calling user's `client_id` without triggering RLS on the users table.

2. **Fixed all client isolation policies** to use `get_my_client_id()` instead of querying the users table directly, eliminating RLS recursion risk:
   - `client_hq_view` on clients
   - `client_isolation_stores` on stores
   - `client_isolation_assets` on assets
   - `client_isolation_jobs` on jobs
   - All vendor, project, region, photo, and maintenance schedule policies

3. **Enabled RLS on `asset_types`** with authenticated SELECT and admin ALL policies.

4. **Added missing write policies** for:
   - Vendors (admin ALL)
   - Projects (admin ALL)
   - Regions (admin ALL + authenticated SELECT)
   - Site photos (admin ALL + client INSERT/DELETE)
   - Asset photos (admin ALL + client INSERT/DELETE)
   - Maintenance schedules (admin ALL + client SELECT)
   - Jobs (authenticated INSERT for fault reporting)

5. **Scoped GRANTs** — removed UPDATE on photo tables (photos are only inserted/deleted).

6. **Analysis page error handling** — added `error` state, extracted `fetchAnalytics` for retry, added user-visible error UI with AlertTriangle icon and "Try Again" button.

## Issues or concerns

- **Policy name conflicts**: The migration uses `DROP POLICY IF EXISTS` for all known policy names from previous migrations. If any policies were manually created with different names, they won't be dropped.
- **Super admin NULL client_id**: `get_my_client_id()` returns NULL for super_admin users, but they're covered by separate `admin_all_*` FOR ALL policies, so client isolation policies won't affect them.
- **Nested RLS evaluation**: Policies on assets/jobs/photos still join through the stores table. This is safe because stores RLS policies use SECURITY DEFINER functions (`get_my_role()`, `get_my_client_id()`) which bypass RLS.
- **Testing needed**: Non-admin users should be tested to verify they can query assets, jobs, and schedules without recursion errors after this migration.
